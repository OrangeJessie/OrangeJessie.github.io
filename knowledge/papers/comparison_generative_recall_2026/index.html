生成式召回近期进展阅读报告

这份报告聚焦推荐/广告/电商里的“生成式召回”：模型不再只做 user/item embedding 后 ANN 检索， 而是把 item 表示成 Semantic IDs 或离散 token 序列，由 Transformer/LLM/diffusion 类模型直接生成候选集合。 本文不逐篇精读，而是给出阅读路线、关键论文简介和工程判断。

一句话结论： 生成式召回已经从“用 Transformer 生成短 Semantic ID”进入到“SID 全生命周期优化”的阶段： 近两年最重要的进展集中在可微/端到端 SID、长 SID 与并行解码、search+recommendation 统一建模、 召回+排序统一、以及工业级 post-training / preference alignment。

主要发展脉络

1. SID 从“离线 tokenizer”变成核心优化对象

TIGER/RQ-KMeans/RQ-VAE 这条线证明了 SID 可行，但 2025-2026 的新工作都在指出同一个问题： SID 如果只为了内容重构或聚类优化，会和推荐目标错位。因此出现了 DAS/DOS/UniSID/DIGER/STAR 等对齐、可微、端到端方法。

2. 解码效率成为召回落地瓶颈

自回归生成 SID 需要 beam search，多 token 长度和多候选召回会放大延迟。 RPG、PinRec、LLaDA-Rec、ACERec 等都在减少串行解码成本，或用并行/扩散/多 token 方式提升多样性和吞吐。

3. 从单一 recommendation 走向 search + recommendation 统一

Spotify/RecSys 线的工作强调：query-driven search 和 query-free recommendation 对 SID 空间的要求不同。 多任务 bi-encoder 或统一 generative backbone 正在成为重要方向。

4. 工业系统开始验证“召回+排序统一”

OneRec、PinRec、TBGRecall、DAS、DOS、Snapchat SID 说明生成式召回不再只是 academic benchmark。 但工业论文也暴露了现实问题：多目标、业务约束、热度偏置、在线收敛、valid ID、冷启动和系统回滚。

实现与研究建议

如果你要做实验复现

从 GRID 或 MiniOneRec 入手最省事：先跑 TIGER/RQ-KMeans/RQ-VAE baseline，再替换 SID tokenizer 或 generative backbone。 不建议一开始就实现端到端 SID，因为 codebook collapse、valid ID 约束和 beam search 成本会同时出现。

如果你要做业务落地

不要只和 ANN/two-tower 比 HitRate。需要同时看召回覆盖、tail item、冷启动、延迟、invalid ID、候选多样性、在线业务指标和回滚方案。 PinRec/OneRec/TBGRecall 的价值在于暴露了这些工程约束。

SID 设计优先级

当前最值得投入的是 SID 对齐而不是单纯换 backbone：把 content semantics、collaborative signals、search signals、generation-space objectives 拉到同一套离散码空间里。

生成范式选择

短 SID + 自回归适合先做 baseline；长 SID/可变长/并行预测适合追求表达力和效率； diffusion-style 方向很新，值得跟踪但工程成熟度还需要验证。

一个重要反方观点

2025 的 scaling-view 论文认为 SID-based GR 在扩大 encoder/tokenizer/RS 规模时可能很快饱和， LLM-as-RS 反而有更好的 model scaling。它提醒我们：SID 是工程上高效的压缩接口，但也可能成为表达瓶颈。

下一步值得跟踪

端到端可微 SID、post-training/RL for generative recommendation、search+rec+ads 多任务统一、工业级 online convergence、以及 codebook debiasing。
